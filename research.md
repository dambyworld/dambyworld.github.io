# GitHub Pages 블로그 계획 리서치

## 문서 목적

- `.blog/github-pages-blog-plan.md`의 계획 내용을 실행 관점에서 상세 해석한 리서치 문서다.
- 이번 단계의 목표는 구현이 아니라, 블로그 구축 방향과 사전 결정 항목을 명확히 정리하는 것이다.
- 대상은 `dambyworld` 계정 기준 개인 기술 블로그이며, 구현과 배포는 보류 상태다.

## 작업 범위 요약

- 포함 범위
  - 블로그 목표 정리
  - 저장소 전략 검토
  - 기술 스택 방향 확인
  - 정보 구조와 운영 정책 초안 정리
  - 구현 전 체크리스트와 보류 항목 정리
- 제외 범위
  - 저장소 생성
  - Next.js 프로젝트 초기화
  - GitHub Pages 설정
  - GitHub Actions 작성
  - 디자인 및 실제 화면 구현
  - 콘텐츠 작성과 배포

## 핵심 목표

- `dambyworld` GitHub 계정으로 운영하는 기술 블로그를 GitHub Pages 기반으로 구축한다.
- 블로그는 개인 기술 아카이브, 실험 기록, 트러블슈팅 회고를 축적하는 용도로 본다.
- 메뉴 구조는 현재 `diary`, `tweet`, `review` 3개 섹션을 중심으로 검토한다.
- 기술 주제는 코프링 (kopring), 프론트엔드, 인프라를 주요 축으로 고려한다.

## 현재 확정에 가까운 방향

### 저장소 전략

- 1순위 권장안은 `dambyworld/dambyworld.github.io` 사용자 페이지 저장소다.
- 이유는 GitHub Pages 사용자 페이지가 루트 도메인 형태로 노출되어 구조가 단순하기 때문이다.
- 메뉴가 3개 정도인 현재 요구사항에서는 저장소를 여러 개로 쪼개기보다 하나의 사용자 페이지 저장소 안에서 섹션 라우팅으로 운영하는 편이 단순하다.
- 별도 저장소 전략은 블로그를 프로필 사이트와 분리하고 싶을 때만 대안으로 본다.

### 도메인/비용 관점

- `github.io` 기본 도메인만 쓰면 추가 비용은 없다.
- 비용이 생기는 경우는 커스텀 도메인을 연결할 때뿐이다.
- 따라서 초기 구축 단계에서는 사용자 페이지 저장소 + 기본 도메인 조합이 가장 낮은 운영 부담을 가진다.

### 기술 스택

- 계획 문서 기준 확정 스택은 Next.js 정적 export다.
- 이유는 React/Next.js 생태계 활용성과 향후 확장성이 높기 때문이다.
- 다만 GitHub Pages는 정적 호스팅이므로, Next.js 기능 중 서버 런타임 의존 기능은 배제하는 방향이 전제된다.

## 기술적 해석

### Next.js 정적 export에서 확인해야 할 점

- App Router를 사용하기로 결정했다.
- 모든 라우트는 정적으로 생성 가능해야 한다.
- 서버 액션, 동적 서버 렌더링, 런타임 의존 API 사용은 배제하는 방향으로 설계한다.
- 콘텐츠 소스는 `.md` 파일 기반 정적 생성 방식으로 결정한다.
- GitHub Pages 배포 시 산출물은 정적 파일로 export되어야 하므로 `output` 설정 검토가 필요하다.

### 저장소 유형에 따른 경로 차이

- 사용자 페이지 저장소(`dambyworld.github.io`)라면 루트 경로 배포가 가능해 경로 설정이 단순하다.
- 프로젝트 페이지 저장소(예: `tech-blog`)라면 `/repo-name` 하위 경로 배포가 되므로 `basePath`, `assetPrefix` 검토가 중요하다.
- 현재 권장안이 사용자 페이지 저장소인 이유 중 하나도 이 경로 복잡도를 줄이기 위해서다.

### 배포 전략

- 권장 배포 방식은 `main` 브랜치에 소스를 두고 GitHub Actions로 Pages 산출물을 배포하는 방식이다.
- 이 접근은 수동 빌드 산출물 관리보다 재현성과 유지보수성이 낫다.
- 구현 단계에 들어가면 Pages 설정 방식과 Actions 권한 범위를 함께 확인해야 한다.

## 요구사항 정리

### 콘텐츠 구조

- 필수 페이지 후보
  - 홈
  - `diary` 목록/상세
  - `tweet` 목록/상세
  - `review` 목록/상세
  - 태그/카테고리
  - About
- 포스트 메타데이터 후보
  - 제목
  - 날짜
  - 설명
  - 태그
  - 카테고리
  - 공개 여부
  - 대표 이미지

### 콘텐츠 디렉터리 방향

- 계획 문서 예시는 `src/content/diary`, `src/content/tweet`, `src/content/review` 구조다.
- 이 구조는 섹션 분리가 명확하고, 각 섹션에 다른 목록 규칙을 둘 때 유리하다.
- 반면 공통 메타 처리와 검색 체계를 생각하면, 실제 구현 시에는 content collection 방식으로 추상화할 여지도 있다.

### 운영 정책 정리

- 작성 언어는 국문/영문 병행으로 운영한다.
- URL 규칙은 날짜형과 slug 중심 방식 중 하나를 확정해야 한다.
- 태그와 카테고리는 동시에 운영한다.
- 섹션별 글 성격은 다음처럼 정의한다.
- `diary`: 비교적 긴 기록형 글
- `tweet`: 짧은 메모/링크형 글
- `review`: 회고/리뷰형 글

### URL 규칙 비교

- 날짜형 URL 예시: `/diary/2026/03/12/github-pages-plan`
- slug 중심 URL 예시: `/diary/github-pages-plan`
- 날짜형은 발행 시점이 주소에 드러나 아카이브 감성이 강하고, 연대기 탐색에 유리하다.
- slug 중심은 주소가 짧고 수정에 유연하며, 콘텐츠가 오래돼도 덜 낡아 보인다.
- 현재 블로그 성격이 기술 아카이브와 회고를 함께 가지므로, 공개 기록성을 강조하면 날짜형이 맞고, 재정리와 재발행 유연성을 중시하면 slug 중심이 맞다.
- 실무적으로는 정렬과 날짜 노출은 UI에서 처리하고, URL은 slug 중심으로 유지하는 방식이 가장 운영 부담이 낮다.

## 디자인 및 운영 관점

### 디자인 방향

- 계획 문서에서는 미니멀, 아카이브형, 매거진형 중 하나를 선택하도록 열어두고 있다.
- 현재 블로그 목적이 "기술 아카이브"에 가깝기 때문에, 탐색성과 가독성을 우선하는 아카이브형이 가장 일관적이다.
- 구현 단계에서는 헤더, 푸터, TOC, 코드블록, 이전/다음 글 네비게이션이 핵심 공통 요소가 된다.

### 미니멀과 아카이브형 차이

- 미니멀은 시각 요소를 줄이고 여백과 단순한 구조에 집중하는 방식이다.
- 아카이브형은 글 탐색, 분류, 목록 구조, 오래된 글 재방문성을 우선하는 방식이다.
- 미니멀은 "적게 보여주는 디자인"에 가깝고, 아카이브형은 "많은 글을 잘 찾게 하는 구조"에 가깝다.
- 둘은 배타적이지 않아서, 시각 톤은 미니멀하게 가져가고 정보 구조는 아카이브형으로 설계하는 혼합 전략이 가능하다.
- 현재 블로그 목적에는 "미니멀한 시각 톤 + 아카이브형 정보 구조"가 가장 자연스럽다.

### 브랜딩 요소

- 사이트 타이틀
- 짧은 소개 문구
- 프로필 정보
- 소셜 링크
- favicon
- OG 이미지 정책

이 항목들은 기술 구현보다 먼저 톤을 정하는 데 영향을 준다.

### 운영 자동화

- 배포 자동화는 GitHub Actions가 기본 선택지다.
- 품질 자동화 후보는 markdown lint, 링크 체크, 이미지 경로 검증이다.
- 분석 도구는 Plausible 또는 Google Analytics가 후보지만, 초기에 반드시 필요한 항목은 아니다.

## 리스크와 결정 포인트

### 구현 전에 먼저 확정해야 하는 것

- 실제 운영 저장소는 사용자 페이지 저장소로 확정한다.
- 구현 작업 디렉터리는 `~/Dev/.blog`로 확정한다.
- Next.js는 App Router로 확정한다.
- 콘텐츠 포맷은 Markdown `.md`로 확정한다.
- `diary`, `tweet`, `review`를 고정 정보 구조로 유지할지

### 잠재 리스크

- Next.js 기능 중 GitHub Pages 정적 export와 맞지 않는 기능을 초기 설계에 섞을 수 있다.
- 저장소 유형을 늦게 바꾸면 경로 설정(`basePath`, asset 경로)을 다시 조정해야 할 수 있다.
- 섹션 정의가 모호하면 정보 구조와 포스트 템플릿이 흔들릴 수 있다.
- 사용자 페이지 저장소와 프로필 사이트 역할이 충돌하면 브랜딩 방향이 애매해질 수 있다.
