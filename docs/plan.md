# Black/White 투톤 디자인 변경 계획

## 1. 작업 목표

- 현재 블로그 디자인의 웜 톤 중심 색감을 Black/White 투톤 계열로 재설계한다.
- 다크모드와 반응형까지 함께 고려한 디자인 시스템 변경 계획을 수립한다.
- 이번 단계에서는 구현하지 않고, 시각 방향과 작업 순서만 문서로 확정한다.

## 2. 현재 상태 진단

현재 글로벌 스타일은 다음 특징을 가진다.

- 배경이 크림/베이지 기반이다.
- 포인트 컬러가 오렌지/브라운 계열이다.
- 카드, 버튼, 링크 hover가 따뜻한 색감 중심이다.
- 밝은 모드만 전제로 잡혀 있고, 다크모드용 토큰 구조는 없다.
- 반응형은 기본 레이아웃 축소 수준으로만 정의되어 있고, 시각 톤 분기까지 고려되진 않았다.

즉, 현재 디자인은 "따뜻한 아카이브"에 가깝고, 이번 변경은 "절제된 Black/White 투톤 에디토리얼" 방향으로 전환하는 작업이다.

## 3. 목표 디자인 톤

### 핵심 방향

- 기본 색상은 검정, 흰색, 중간 회색만 사용한다.
- 포인트 색은 원칙적으로 두지 않거나, 아주 약한 gray scale 대비만 사용한다.
- 화면 인상은 "조용함", "선명한 대비", "에디토리얼", "아카이브" 쪽으로 맞춘다.
- 장식적 따뜻함보다 타이포그래피와 레이아웃 리듬으로 개성을 만든다.

### 시각 키워드

- monochrome
- editorial
- archival
- quiet contrast
- sharp spacing

## 4. 변경 대상

### 색상 토큰

현재 `app/globals.css`의 다음 토큰은 전면 재정의 대상이다.

- `--bg`
- `--surface`
- `--surface-strong`
- `--text`
- `--muted`
- `--line`
- `--accent`
- `--accent-soft`
- `--accent-deep`
- `--shadow`

### 컴포넌트 스타일

- 헤더
- 네비게이션 pill
- 홈 hero
- section panel
- post card
- sidebar card
- 버튼 계열
- 링크 hover
- 코드 블록
- footer

### 레이아웃/상호작용

- 데스크톱에서의 카드 대비감
- 모바일에서의 여백 압축 방식
- hover/focus 시 컬러 변화 대신 명도/테두리/위치 변화 사용
- 다크모드 전환 시 배경과 텍스트 대비 유지

## 5. 제안 토큰 시스템

### 라이트 모드

- `--bg`: 거의 흰색 계열
- `--surface`: 순백에 가까운 카드 배경
- `--text`: 거의 검정에 가까운 본문색
- `--muted`: 중간 회색
- `--line`: 연한 회색 보더
- `--shadow`: 매우 얕은 검정 그림자

### 다크 모드

- `--bg`: 거의 검정에 가까운 배경
- `--surface`: 짙은 회색 카드 배경
- `--text`: 거의 흰색에 가까운 본문색
- `--muted`: 밝은 회색
- `--line`: 어두운 회색 또는 반투명 밝은 보더
- `--shadow`: 거의 사용하지 않거나 매우 약하게 유지

### 포인트 처리 원칙

- 기존 `accent` 개념은 삭제하거나 gray scale 강조값으로 축소한다.
- 강조는 채색이 아니라 두께, 반전, underline, border contrast로 해결한다.
- CTA 버튼도 검정 바탕/흰 글자 또는 반전 스타일만 사용한다.

## 6. 다크모드 지원 계획

### 적용 방식

- CSS custom property 기반으로 라이트/다크 테마를 분리한다.
- `prefers-color-scheme`를 기본으로 지원한다.
- 필요하면 이후 단계에서 수동 토글을 추가할 수 있게 토큰 구조를 먼저 정리한다.

### 구현 시 고려사항

- 배경과 카드가 동시에 너무 붙지 않도록 명도 차를 최소 2단계 이상 둔다.
- 본문 텍스트와 muted 텍스트의 대비를 WCAG 기준에 가깝게 유지한다.
- 코드 블록은 라이트/다크에서 각각 별도 대비 세트를 둔다.
- hover/focus 상태가 다크모드에서도 확실히 드러나야 한다.

## 7. 반응형 지원 계획

### 모바일

- 상단 헤더 패딩과 카드 radius를 더 단단하게 줄인다.
- pill 네비게이션은 줄바꿈 기준을 명확히 하고 터치 면적을 유지한다.
- hero 영역은 한 컬럼으로 단순화한다.
- 텍스트 대비는 유지하되, 회색 단계가 너무 많아 흐려 보이지 않게 제한한다.

### 태블릿

- 섹션 카드 2단 또는 1단 전환 기준을 명확히 한다.
- 사이드바는 sticky 유지 여부보다 읽기 흐름을 우선한다.

### 데스크톱

- 넓은 여백과 큰 타이포 대비를 유지한다.
- 흑백 톤에서 밋밋해지지 않도록 굵기, 간격, 분할선 리듬을 강화한다.

## 8. 구현 순서 제안

1. `app/globals.css`의 색상 토큰을 monochrome 구조로 재정의한다.
2. 버튼, 네비게이션, 카드 hover를 색상 변화 중심에서 border/contrast 중심으로 바꾼다.
3. 라이트 모드 기준 전체 화면 톤을 정리한다.
4. `prefers-color-scheme: dark` 기준 다크모드 토큰을 추가한다.
5. 모바일/태블릿/데스크톱 breakpoint별 밀도와 여백을 다시 조정한다.
6. 코드 블록과 링크 가독성을 별도 점검한다.
7. 최종적으로 홈, 목록, 상세 페이지를 화면별로 확인한다.

## 9. 검토 체크리스트

- 라이트 모드가 완전히 흑백 중심으로 정리됐는가
- 다크모드에서 텍스트 대비가 충분한가
- 버튼과 링크 상태가 색상 없이도 구분되는가
- 모바일에서 pill 네비게이션이 과밀하지 않은가
- 상세 페이지 본문과 코드 블록 가독성이 유지되는가
- 홈 hero와 카드 구조가 밋밋하지 않은가

## 10. 보류 사항

- 실제 CSS 토큰 수정
- 다크모드 토글 UI 추가 여부 결정
- 스크린샷 비교 검증
- 최종 시각 QA

## 11. 결론

- 현재 디자인은 웜 톤 아카이브형이고, 목표 디자인은 Black/White 투톤 에디토리얼 아카이브형이다.
- 구현 시 핵심은 색을 바꾸는 것보다 "대비 구조"를 다시 설계하는 데 있다.
- 다음 구현 단계에서는 라이트/다크 공통 토큰 체계를 먼저 만들고, 그 위에 반응형 밀도 조정을 올리는 방식이 가장 안전하다.
